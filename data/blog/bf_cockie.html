Brute-forcing a Stay-Logged-In Cookie

Tags: Authentication, Cookies, Brute Force, MD5, BurpSuite

🎯 Goal

Exploit weak stay-logged-in cookies to gain access to Carlos’s account by brute forcing the cookie value.

🛠️ Walkthrough

1. Capturing the Cookie

After enabling “Stay logged in”, the request contained 2 tokens: one looked random, the other was Base64 encoded. The encoded one was sent to Burp Decoder.

2. Analyzing the Token

Decoded value looked like username:password, but the password part was hashed. Its length was 32 chars, suggesting MD5 hash.

3. Preparing the Attack

Configured Burp Intruder to brute force the MD5 hash of Carlos’s password using the provided wordlist.

  • Payload: Carlos’s password candidates
  • Rules: Add Prefix 'carlos:' → Hash MD5 → Base64 encode
  • Injected into the cookie parameter

4. Running the Attack

Sorted the responses by status code. Found one request returning 200 OK. This indicated the valid token for Carlos’s session.

5. Lab Solved

Successfully brute forced Carlos’s stay-logged-in cookie and gained access to his account ✅

💡 Lessons Learned

  • Never store static hashes (like MD5) in cookies.
  • Stay-logged-in functionality must use strong, random, server-side validated tokens.
  • Burp Suite Intruder with transformations (MD5 + Base64) is powerful for automating attacks.
