DOM XSS in document.write using location.search

Tags: XSS, DOM, document.write, location.search, client-side

Exploit a DOM-based XSS vulnerability caused by unsafe usage of document.write with location.search inside a <select> element.

1. Analyze Functionality

The stock checker feature builds a <select> dropdown using the storeId parameter from the URL. The value is directly written using document.write, making it controllable.

2. Observation

If we pass a custom storeId value in the query string, it appears inside the <select> tag. Example:

?storeId=test

This results in:

<select><option>test</option></select>

3. Crafting the Payload

We can break out of the attribute, close the <select> tag, and inject malicious HTML/JS. For example:

storeId="></select><img src=0 onerror=alert('m4zix')>

4. Execution

When loading the page with the payload, the <img> tag is rendered and the onerror event fires, executing our JavaScript payload:

alert('m4zix')

5. Lab Solved

The alert box popped up successfully ‚úÖ DOM-based XSS confirmed.

üí° Lessons Learned

  • document.write is dangerous when writing user-controlled input.
  • Always sanitize and escape data before inserting it into the DOM.
  • Prefer using safe DOM APIs (textContent, createElement) over string concatenation.
  • Client-side code can be just as vulnerable as server-side code.
