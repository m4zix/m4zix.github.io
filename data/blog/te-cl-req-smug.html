HTTP Request Smuggling - TE.CL

Tags: HTTP, Smuggling, TE.CL, Security Testing

🎯 Goal

Confirm a TE.CL HTTP request smuggling vulnerability via differential responses and deliver a smuggled request.

🛠️ Walkthrough

1. Identify Header Behavior

Determine whether frontend/backend rely on Content-Length (CL) or Transfer-Encoding (TE) headers. Use HTTP/1.1 requests for testing.

2. Differential Testing

  • Send request with incorrect large Content-Length + correct body: CL.TE → 400 Bad Request error.
  • Send request with incorrect large Content-Length + correct body: TE.CL → 500 Timeout error.

Observed response → frontend uses TE → TE.CL vulnerability confirmed.

3. Smuggle Request

Craft payload to smuggle an HTTP request through the TE.CL processing:

POST / HTTP/1.1
Host: target.com
Content-Length: 4
Transfer-Encoding: chunked

2c
POST /404 HTTP/1.1
Content-Length:15

x=1
0

Sending a normal request afterwards will be corrupted by the smuggled request → confirms successful exploitation ✅

💡 Lessons Learned

  • TE.CL vulnerabilities arise when backend processes both Transfer-Encoding and Content-Length inconsistently.
  • Differential responses (400 vs 500) help identify the vulnerable header sequence.
  • Crafted smuggled requests can bypass frontend checks and manipulate backend processing.
  • Always test with HTTP/1.1; HTTP/2 handles headers differently and may prevent smuggling.
